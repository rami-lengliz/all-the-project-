# Chatbot Foundation Architecture

## What was added
- Two Prisma database models: `ChatConversation` and `ChatMessage`
- A modular NestJS capability structured in `src/chatbot/`:
  - `ChatbotModule`
  - `ChatbotController`
  - `ChatbotService`
  - `ChatbotMemoryService`
  - `ChatbotOrchestratorService`
  - `ChatbotToolsService`
- Endpoints representing authenticated user creation and retrieval of chatbot memory.
- An internalized, highly constrained AI execution loop powered by OpenAI.
- One strict tool: `search_listings`.

## Architecture Flow
1. **User Request**: The frontend calls `POST /chatbot/messages` with an optional `conversationId`.
2. **Memory Service**: Persists the user message locally inside PostgreSQL to guarantee an immutable trail and enforces strict ownership. 
3. **Orchestrator Service**: Receives the new message, retrieves the last 15 messages of context, and precisely reconstructs state. It intelligently recreates the schema bindings mapping historically invoked operations to their strictly formatted `tool_call_id`s, ensuring zero format rejections from OpenAI APIs.
4. **Tool Loop (Optional)**: If OpenAI invokes `search_listings`, `ChatbotToolsService` dynamically maps categorical nomenclature bounds down to precise system slugs (e.g. `accommodation` -> `stays`). All parsed variables (`args`) undergo type assertion (`typeof args === 'object'`). The loop now robustly handles **multiple concurrent tool calls** generated by OpenAI dynamically, executing and persisting them sequentially before synthesizing a final summarization callback.
5. **Response**: Output merges seamlessly via `TransformInterceptor` into the mandatory `{"success": true, "data": ..., "timestamp": "ISO"}` API payload constraint.

## Endpoints
All endpoints enforce `JwtAuthGuard` and standard data wrapper intercepts.
- `POST /chatbot/messages` - Send a message to the AI or spin up a new conversation context constraint.
- `GET /chatbot/conversations` - Retrieve an overview of historically owned sessions.
- `GET /chatbot/conversations/:id/messages` - Trace granular memory steps strictly bounded to the invoker identity.

## Tool Behavior
- **`search_listings`**: The solo operational tool explicitly mapped. It prevents hallucinatory variables by enforcing: 
  - `category` securely bounded inside `[accommodation, mobility, water-beach-activities]` and mapped identically (`slugMap`) ensuring strict sync with underlying schemas.
  - Type-safe payload validation catching broken JSON outputs.
  - `radiusKm` relies on an explicitly clamped interval of 1 to 50 bounding. Default implies 10km proximity fallback. 
  - Never accesses PostgreSQL raw statements; purely binds its input to `listingsService.findAll()`.

## Security Constraints
- **Resource Ownership**: Conversations are explicitly tethered to User IDs (`UUID`). Memory fetching or writing halts execution immediately passing 403 Forbidden boundaries if cross-account contamination attempts occur.
- **Input Sanitization**: `CreateMessageDto` drops inbound queries surpassing 1000 bounds.
- **Data Protection**: Output execution ignores pseudo-functions hallucinated by the model; strict internal matching to `ALLOWED_CATEGORIES` drops toxic payloads gracefully.
- **Financial Segregation**: Zero linkage or references internally to booking, ledgers, payouts, or mutating states. Purely read-only interactions.

## Test Coverage
Automated E2E tests target `/test/chatbot.e2e-spec.ts`. Scopes secured:
- First-time message spins a contextualized identity.
- Messages correctly bind and append securely onto subsequent interactions.
- Accessing cross-origin peer conversations forces rejection.
- Geographic constraints cleanly default `radiusKm` attributes and lock at a 50 km bounds ceiling safely.
- Follow-up query violations trigger embedded prompt override clauses limiting chat latency effectively.
